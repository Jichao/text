Examples

Hello, World!

Ok, here is the obligatory "Hello, world!" example: 

#include <boost/text/text.hpp>

#include <iostream>


int main ()
{
    boost::text::text_view const text_view("Hello, world!\n");
    boost::text::text const text("всем привет!\n");

    std::cout << text_view;
    std::cout << text;

    return 0;
}

Pretty simple, we've just made a non-owning text_view, and an owning-and-allocating text. They each say "Hello, World!". The second one is what the Internet assures me is idiomatic Russian meaning the same thing.

Establishing Some Guarantees

If we want to ensure that we do all our Unicode processing safely, we just need to establish the invariant that our string is properly encoded: 

// We can start with a literal or other user-verified encoded string.
boost::text::text_view tv = "всем привет!\n";

// Or we can explicitly run over the entire string and check its encoding.
boost::text::text t = boost::text::checked_encoding(string_from_terminal());

And then all subsequent operations that use the safe interfaces (i.e. not the non-const index operator or mutable iterators), we are guaranteed that the string remains encoded properly: 

try {
    // If we try to slice the encoding, we get an exception.
    boost::text::text_view slice_0 = tv(0, 1);
    assert(!"We should never evaluate this assert.");
} catch (...) {}

// But as long as we slice along code point boundaries, everything works.
boost::text::text_view slice_1 = tv(0, 2);

Breaking Encoding on Purpose

Of course, we may wish to break the encoding for some reason. That too is allowed, by using the unsafe interfaces: 

boost::text::text text_slice_0(tv.begin(), tv.begin() + 1);
boost::text::text text_slice_1(tv.begin() + 1, tv.end());

std::cout << text_slice_0 << "\n"; // prints "?\n" or some other garbage indicator

The breakage can also be repaired again later: 

text_slice_0.insert(
    text_slice_0.end(),
    text_slice_1.begin(), text_slice_1.end()
);

std::cout << text_slice_0; // prints "всем привет!\n"

Using Text Types in Interfaces

There are particular parameter passing conventions that some of the Boost.Text types are designed for. For instance, text_view, like std::string_view, is designed to bind to any char const *, text_view, or text argument, without making a copy. text_view is therefore a replacement for the multiple overloads it would take to handle all those types. If text_view was not implicitly convertible from char const * and text, this is the sort of code you would have to write: 

void print_string_without (boost::text::text_view str)
{ std::cout << str; }

void print_string_without (boost::text::text const & str)
{ print_string_without(boost::text::text_view(str)); }

void print_string_without (char const * str)
{ print_string_without(boost::text::text_view(str)); }

And here is what you can write with text_view's implicit conversions: 

void print_string_with (boost::text::text_view str)
{ std::cout << str; }

If you want to write a function parameter that handles any of Boost.Text's types, rope_view is available: 

void print_any_text_type (boost::text::rope_view str)
{ std::cout << str; }

And, sure enough, you can call it with any of the Boost.Text types, and even char const *: 

print_any_text_type("1\n");
print_any_text_type(boost::text::text_view("2\n"));
print_any_text_type(boost::text::text("3\n"));
print_any_text_type(boost::text::repeat("4\n", 4));
print_any_text_type(boost::text::rope("5\n"));

Note

rope_view is less efficient than text_view for binding to texts and text_views (though only a little). If you know you need that extra efficiency, and you know you don't need to support ropes or repeated_text_views as arguments to your function, stick to text_view.

Ropes in Practice

A key use of ropes is in text editors. Every text editor you've ever used that handles large files gracefully has done something to break the file into chunks -- editing the beginning of a 64MB file that is stored in a contiguous sequence in memory is way too slow. The use of a rope is a common approach to remedy this.

In the example directory, there are two programs, text_editor and rope_editor. The first uses an in-memory text object to store text that is edited. It is unusably slow. The second does nearly the same thing, but uses a rope. It is way more usable.

Note

These editors are pretty lame. They don't have cut/copy/paste, and can't even save your file! They use Emacs keybindings for what little they do. They use the ncurses library, which is apparently pretty buggy on MacOS.
