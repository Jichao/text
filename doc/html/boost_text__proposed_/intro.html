Intro

"It's all too much and not enough at the same time."

-- Jack Kerouac, referring to std::string

Motivation

std::string is an infamous design. It's fat -- why are there so many member functions? It's not fat enough -- what if I need to support non-ASCII characters? I could use std::basic_string<CharT> for some CharT that works for non-ASCII uses, but then I'm left using multiple types for strings, or making everything a template parameterized on CharT.

This library is an attempt to re-think std::string as an interoperable family of types, each suited to a different aspect of string processing.

Features

- text_view, a std::string_view replacement that is: constexpr, optionally null-terminated; UTF-8 encoded; and immutable.

- text, a std::string replacement that is: always null-terminated; strongly exception safe; UTF-8 encoded; and mutable.

- rope, a Rope-like type with copy-on-write semantics that is: non-null-terminated; basically exception safe; UTF-8 encoded; and mutable.

- rope_view, an immutable non-owning view of a rope, text, text_view, or repeated_text_view.

- repeated_text_view, an immutable utility type representing a repeated text_view.

- Substrings of each of the five types above can be had with no copies or allocations using a Python-like slicing interface.

- Compilational simplicity -- none of the above five types is a template.

- Iterators converting UTF-8 to and from UTF-16 and UTF-32.

- A small library of algorithms that replicates the functionality of std::string's search members, plus a bit more.
